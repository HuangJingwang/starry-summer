# Inline Selection Comments Design

## Goal

Add Feishu-style inline comments to public article reading pages. Readers can select text in an article, create a comment for that exact passage, and later see approved comments as highlighted text with a right-side discussion rail.

## Scope

This feature extends the existing comments system instead of creating a separate annotation product. Inline comments are ordinary comments with optional selection anchor metadata. Existing full-article comments remain available.

V1 includes:

- Text selection inside rendered article bodies.
- A small floating comment action next to the selection.
- A desktop right-side inline comment rail.
- A mobile bottom drawer for the same comment rail.
- Approved inline comments rendered as highlighted passages in the article body.
- Click-to-focus behavior between highlighted text and the comment rail.
- GitHub reader login and existing moderation flow for submissions.
- Admin moderation visibility for the selected passage.

V1 excludes:

- Reply threads under one anchor.
- Resolve and reopen states.
- Real-time collaboration.
- Rich-text comment bodies.
- Advanced anchor recovery after large article rewrites.

## User Experience

Desktop uses the selected visual option A:

1. A reader selects text inside `.detail__body`.
2. A compact floating `评论` action appears near the selection.
3. If the reader is not logged in, clicking the action opens the same GitHub login gate used by article comments.
4. If logged in, the right-side inline comment rail opens.
5. The rail shows the selected passage, a textarea, and a submit action.
6. Submitting creates a pending comment. The reader sees a pending/audit message.
7. Approved inline comments render as soft yellow highlights in the article body.
8. Clicking a highlight focuses the matching comment card in the rail.
9. Clicking a comment card scrolls or focuses the matching highlight in the article body.

Mobile uses the same selection and highlighting behavior, but the rail is presented as a bottom drawer. This avoids shrinking the reading column and keeps the interaction reachable on narrow screens.

## Data Model

The existing `comments` table gains optional anchor fields:

- `anchor_text text`
- `anchor_prefix text`
- `anchor_suffix text`
- `anchor_start integer`
- `anchor_end integer`
- `anchor_hash text`

Rules:

- Plain article comments have all anchor fields empty.
- Inline comments must include `anchor_text`, `anchor_start`, `anchor_end`, and `anchor_hash`.
- `anchor_start` and `anchor_end` refer to offsets in the article body's normalized text content, not raw Markdown.
- `anchor_prefix` and `anchor_suffix` store short surrounding text snippets for future recovery and admin context.
- `anchor_hash` is derived from normalized selected text plus surrounding context. It is used to group comments that target the same passage.

The API record shape extends `CommentRecord` and public/admin comment DTOs with optional anchor metadata:

```ts
interface CommentAnchor {
  text: string;
  prefix: string;
  suffix: string;
  start: number;
  end: number;
  hash: string;
}
```

## Backend API

The existing `POST /comments` endpoint accepts an optional `anchor` object.

Validation:

- `anchor.text` must be non-empty and at most 500 characters.
- `anchor.prefix` and `anchor.suffix` are each at most 160 characters.
- `anchor.start` and `anchor.end` must be finite integers, with `0 <= start < end`.
- `anchor.end - anchor.start` must be at most 1,000 characters.
- `anchor.hash` must be a stable lowercase hex string generated by the client helper or recalculated server-side from the normalized anchor.
- Anchored comments are allowed only for the same targets as normal comments: `post`, `note`, and `project`.

Moderation:

- Inline comments enter the same pending queue as regular comments.
- Admin comment cards show the selected passage when anchor metadata exists.
- Approved inline comments are returned by `GET /comments/:targetType/:targetId` with anchor metadata.

## Frontend Architecture

New frontend units:

- `apps/web/src/lib/selection-comments.ts`
  - Builds normalized text anchors from the DOM selection.
  - Computes hash and context snippets.
  - Maps approved comment anchors back onto rendered article text.
  - Provides deterministic pure functions for tests.

- `apps/web/src/components/InlineCommentLayer.tsx`
  - Client component mounted by `ContentDetail`.
  - Receives `targetType`, `targetId`, `reader`, `loginNextPath`, and approved comments.
  - Listens for selections inside the article body.
  - Shows the floating action.
  - Renders the rail/drawer and submission form.
  - Applies highlight spans after the Markdown body is rendered.

Existing units to modify:

- `ContentDetail.tsx`
  - Loads approved comments once.
  - Splits normal article comments and anchored inline comments for presentation.
  - Mounts `InlineCommentLayer` around or near `.detail__body`.

- `interaction-client.ts`
  - Adds `CommentAnchorInput`.
  - Extends `buildCommentRequest` to include optional anchor metadata.

- `CommentForm.tsx`
  - Remains the full-article comment form.
  - Does not handle selection comments directly.

- `styles.css`
  - Adds highlight, floating action, rail, drawer, selected card, and pending message styles.

## Anchor Mapping

V1 uses a practical deterministic strategy:

1. Normalize whitespace in article body text and selected text.
2. Store the selected normalized text, prefix, suffix, and offsets at submission time.
3. When rendering approved inline comments, find the first matching text range using:
   - exact offset if the body text at that range still matches;
   - otherwise exact text plus matching prefix/suffix;
   - otherwise exact text only.
4. If no match is found, keep the comment visible in the rail with an `原文已变更` state and skip body highlighting.

This keeps the first version reliable for normal edits without overbuilding a collaborative document anchoring engine.

## Error Handling

- Empty selections do not show the floating action.
- Selections outside `.detail__body` are ignored.
- Multi-block selections are accepted if their normalized text is within the limit.
- If the reader is not logged in, the rail shows the login gate.
- If submission fails, the rail keeps the draft and shows an error message.
- If an approved anchor cannot be mapped, the comment remains visible in the rail with a changed-source label.
- If overlapping highlights exist, V1 renders the earliest/longest stable ranges and still lists all comments in the rail.

## Privacy And Moderation

Inline comments use the existing GitHub reader identity requirement. They also use the existing IP hash, user agent metadata, rate limit guard, pending status, and admin moderation flow. Public responses strip moderation metadata exactly as normal comments do.

## Testing Strategy

Backend tests:

- Service validates inline anchor payloads.
- Repository insert/select maps anchor columns for in-memory and PostgreSQL repositories.
- Public list returns anchor data for approved comments.
- Admin list includes anchor data for moderation cards.
- Migration tests include the new columns.

Frontend tests:

- `selection-comments` builds anchors from selected text and context.
- Anchor mapping finds exact offset matches.
- Anchor mapping falls back to text/context after small edits.
- Unmapped anchors produce a rail-only state.
- `buildCommentRequest` includes optional anchor metadata.
- `ContentDetail` mounts `InlineCommentLayer` for commentable content.
- Source tests confirm inline rail/drawer classes and admin selected passage labels exist.

Manual verification:

- Select a passage on a post while logged out and see the login gate.
- Select a passage while logged in and submit an inline comment.
- Approve the comment in admin moderation.
- Reload the public article and see the highlighted passage and comment rail.
- Check mobile width and confirm the rail becomes a bottom drawer.
